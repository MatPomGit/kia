1. Cele i rezultaty uczenia się

Znaczenie praktyczne. Git rejestruje historię zmian w plikach, umożliwia cofanie się do wcześniejszych wersji i wspiera równoległą pracę wielu osób. Jest podstawowym narzędziem w projektach programistycznych, badawczych i dokumentacyjnych.

2. Konfiguracja i identyfikacja stanowiska

Tożsamość autora. Każdy commit zawiera nazwę i adres e-mail autora. Dane te powinny identyfikować osobę wykonującą zmianę, ponieważ stają się częścią historii repozytorium.

2.1. Sprawdzenie konfiguracji

git config --global user.name
git config --global user.email

2.2. Ustawienie danych

git config --global user.name "Imię Nazwisko"
git config --global user.email "adres@example.com"
Bez opcji --global ustawienie obowiązuje tylko w bieżącym repozytorium. Jest to dobre rozwiązanie, gdy dla różnych projektów używa się różnych adresów e-mail.

2.3. Numer stanowiska

echo "$HOST"
hostname

2.4. Adresy repozytoriów laboratoryjnych

xx oznacza numer grupy, a yy numer stanowiska.

2.5. Wariant bez dostępu do serwera

mkdir -p "$HOME/cw6/repo"
cd "$HOME/cw6/repo"
git init --bare .

Repozytorium bare. Repozytorium typu bare nie ma katalogu roboczego. Służy jako centralny punkt wymiany zmian, podobnie jak repozytorium na serwerze.

3. Inicjowanie repozytorium

Trzy obszary Git. Git rozróżnia katalog roboczy, indeks oraz bazę commitów. Zmiana wykonana w pliku nie trafia automatycznie do historii — najpierw musi zostać dodana do indeksu.

3.1. Utworzenie repozytorium

mkdir -p "$HOME/cw6/pryw"
cd "$HOME/cw6/pryw"
git init .

3.2. Dodanie pierwszego pliku

cp /sciezka/do/raport.tex .
git add raport.tex
git commit -m "Dodanie początkowej wersji raportu"

3.3. Sprawdzenie stanu

git status
Komunikat working tree clean oznacza, że katalog roboczy nie zawiera zmian względem ostatniego commitu.

4. Historia, status i tagi

Commit jako punkt historii. Commit zapisuje stan przygotowanych plików wraz z autorem, datą, opisem i identyfikatorem. Nie jest kopią całego projektu w sensie użytkowym, lecz elementem grafu historii.

4.1. Kolejna zmiana

git add raport.tex
git commit -m "Aktualizacja autora i dodanie sekcji Nowość"

4.2. Historia

git log
git log --oneline --graph --decorate --all

SHA. Każdy commit ma unikatowy skrót kryptograficzny. W codziennej pracy zwykle wystarcza jego początkowy fragment, o ile jednoznacznie wskazuje commit.

4.3. Tag

git tag MojaEtykieta
git show MojaEtykieta

4.4. Commit wszystkich śledzonych zmian

git commit -am "Kolejna zmiana raportu"
Opcja -a obejmuje tylko pliki już śledzone. Nowe pliki nadal wymagają wcześniejszego git add.

5. Praca lokalna i klonowanie

Klon. git clone tworzy nowe repozytorium zawierające historię, gałęzie i konfigurację zdalnego źródła. Nie jest to zwykłe kopiowanie katalogu.

mkdir -p "$HOME/cw6/lok"
cd "$HOME/cw6/lok"
git clone "file://$HOME/cw6/pryw" .

5.1. Dodanie kodu C

mkdir -p kod

cat > kod/start.c <<'EOF'
#include <stdio.h>

int main(void)
{
    puts("Hello world");
    return 0;
}
EOF

git add kod/start.c
git commit -m "Dodanie programu Hello world"
Kropka na końcu git clone ... . oznacza klonowanie bezpośrednio do bieżącego katalogu.

6. Repozytoria zdalne

Remote jako alias. Nazwa origin lub serwer jest lokalnym skrótem adresu repozytorium. Można zdefiniować wiele różnych zdalnych źródeł.

6.1. Lista zdalnych repozytoriów

git remote -v

6.2. Dodanie repozytorium

git remote add serwer REPO_PRYWATNE

6.3. Pierwsze wysłanie gałęzi

git push --set-upstream serwer main

Upstream. Ustawienie gałęzi śledzonej pozwala później używać skróconych poleceń git push i git pull bez każdorazowego podawania remote i nazwy gałęzi.

W starszych repozytoriach domyślna gałąź może nazywać się master. Sprawdź ją poleceniem git branch --show-current.

7. Synchronizacja zmian

Pobieranie i scalanie. fetch pobiera historię ze zdalnego repozytorium, ale nie zmienia katalogu roboczego. merge integruje pobraną historię z bieżącą gałęzią.

7.1. Pobranie zmian

git fetch origin
git merge --ff-only origin/main

Fast-forward. Scalanie typu fast-forward przesuwa wskaźnik gałęzi do przodu bez tworzenia dodatkowego commitu, gdy lokalna historia nie zawiera własnych rozbieżnych zmian.

7.2. Zmiana i wysłanie

git commit -am "Dodanie nowej sekcji raportu"
git push

7.3. Polecenie pull

git config pull.rebase false
git pull
git pull jest skrótem dla pobrania zmian i ich integracji. W zależności od konfiguracji może wykonywać merge albo rebase.

8. Konflikty scalania

Kiedy powstaje konflikt? Git potrafi automatycznie połączyć zmiany, jeśli dotyczą różnych fragmentów pliku. Konflikt pojawia się, gdy dwie gałęzie zmieniają ten sam obszar w niezgodny sposób.

8.1. Znaczniki konfliktu

<<<<<<< HEAD
Wersja lokalna
=======
Wersja z repozytorium zdalnego
>>>>>>> origin/main

8.2. Procedura rozwiązania

  1. Otwórz każdy skonfliktowany plik.
  2. Wybierz poprawną treść lub połącz obie wersje.
  3. Usuń znaczniki konfliktu.
  4. Dodaj poprawiony plik do indeksu.
  5. Zatwierdź wynik scalania.
git status
git add raport.tex
git commit -m "Rozwiązanie konfliktu w raporcie"
git push
Nie zatwierdzaj pliku zawierającego znaczniki <<<<<<<, ======= i >>>>>>>. Oznacza to, że konflikt nie został rozwiązany.

9. Praca zespołowa

Wspólna historia. Repozytorium zespołowe powinno zawierać logicznie podzielone zmiany, czytelne opisy commitów i strukturę plików minimalizującą ryzyko konfliktów.

9.1. Klonowanie repozytorium wspólnego

mkdir -p "$HOME/cw6/wsp"
cd "$HOME/cw6/wsp"
git clone REPO_WSPOLNE .

9.2. Własny rozdział

mkdir -p NUMER_INDEKSU

cat > NUMER_INDEKSU/tresc.tex <<'EOF'
\chapter{Mój tytuł rozdziału}
\section{Nazwa punktu}

A tu można wstawić dowolny tekst lub rysunek.
EOF

W pliku głównym dodaj:

\include{NUMER_INDEKSU/tresc.tex}

9.3. Zatwierdzenie

git add wspolna.tex NUMER_INDEKSU/tresc.tex
git commit -m "Dodanie rozdziału NUMER_INDEKSU"
git pull
git push
Przed wysłaniem zmian wykonaj git pull. Zmniejsza to ryzyko odrzucenia operacji push z powodu nowszych zmian na serwerze.

10. Gałęzie i scalanie

Gałąź jako wskaźnik. Gałąź nie jest osobną kopią całego projektu. Jest ruchomym wskaźnikiem na commit, który umożliwia rozwój funkcji bez bezpośredniej modyfikacji głównej linii historii.

10.1. Utworzenie gałęzi

git switch -c vNUMER_INDEKSU

Alternatywa zgodna ze starszymi wersjami Git:

git checkout -b vNUMER_INDEKSU

10.2. Commit i push gałęzi

git commit -am "Rozwinięcie własnego rozdziału"
git push --set-upstream origin vNUMER_INDEKSU

10.3. Scalenie z gałęzią główną

git switch main
git pull
git merge vNUMER_INDEKSU
git push

10.4. Usunięcie lokalnej gałęzi

git branch -d vNUMER_INDEKSU
Opcja -d usuwa gałąź tylko wtedy, gdy została scalona. Opcja -D wymusza usunięcie i powinna być używana ostrożnie.

11. Diagnostyka i analiza historii

Ślad audytowy. Git pozwala sprawdzić nie tylko bieżący stan, ale również autora każdej linii, strukturę gałęzi i zawartość repozytorium zdalnego.

11.1. Autorzy linii

git blame wspolna.tex
git blame NUMER_INDEKSU/tresc.tex

11.2. Gałęzie lokalne i zdalne

git branch
git branch -r
git branch -a

11.3. Referencje na serwerze

git ls-remote origin

11.4. Czytelny graf historii

git log --graph --oneline --decorate --all

13. Zakończenie ćwiczenia

Archiwizacja. Repozytorium zdalne powinno zawierać wszystkie wymagane commity. Kopiowanie wyłącznie plików roboczych nie zachowuje historii Git.

  1. Sprawdź stan repozytoriów poleceniem git status.
  2. Upewnij się, że wszystkie wymagane zmiany zostały wysłane.
  3. Zapisz adres repozytorium oraz nazwę ostatniego commitu lub tagu.
  4. Po akceptacji prowadzącego usuń katalog ćwiczeniowy.
rm -rf "$HOME/cw6"
Przed usunięciem sprawdź, czy ostatnie zmiany są widoczne w repozytorium zdalnym. Niewysłane commity zostaną utracone razem z katalogiem lokalnym.
